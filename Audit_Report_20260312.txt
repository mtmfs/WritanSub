WritanSub 项目代码审计报告 (2026-03-12)
=======================================

1. 核心架构风险：ResourceRegistry (资源管理注册表)
------------------------------------------------
- [高危] 模型生命周期管理 (flush_models): 
  目前的实现是在 batch 结束后强制清空所有模型。如果 GUI 中有多个任务并发执行，一个任务的结束会直接导致另一个任务因模型被卸载而崩溃。
- [隐患] faster-whisper 释放逻辑 (_ct2_exit_guard): 
  仅保留上一个模型的 guard 逻辑在频繁切换模型或多实例时可能不够稳健，仍有触发 CTranslate2 崩溃的风险。
- [改进建议]: 引入引用计数或“最后使用时间”逻辑，由注册表自动平衡显存，而不是由 Pipeline 强行干预。

2. GUI 稳定性风险：线程安全与 UI 更新
------------------------------------
- [致命] 非线程安全 UI 更新: 
  pipeline_tab.py 在后台线程中直接调用了 self._log_msg 等 UI 更新方法。在 Qt 架构中，子线程直接操作 QWidget 会导致随机崩溃、界面冻结或 Segfault。
- [改进建议]: 必须通过 QtCore.Signal 发送数据，由主线程的 Slot 进行 UI 渲染。

3. 模型推理效率：TIGER 分块处理 (_chunk_inference)
------------------------------------------------
- [性能] 内存/显存“伪分块”: 
  代码中虽然实现了分块逻辑，但在推理前使用了 torch.cat 将所有分块拼回了一个巨大的 Batch。这完全失去了分块降低显存占用的意义，长音频依然会触发 OOM。
- [改进建议]: 改为微批次 (Micro-batch) 推理，在循环内逐个或按小 Batch 处理。

4. 业务逻辑风险：Pipeline 与 Whisper
-----------------------------------
- [隐患] 重叠区域处理: 
  对每个重叠片段都创建临时文件并初始化一次推理，在重叠较多时 IO 开销极大。
- [隐患] Whisper 版本硬编码: 
  硬编码使用 large-v3。这会导致低配置显卡用户无法运行，且无法根据需求切换更快的 base/small 模型。
- [风险] 异常传播: 
  流水线步骤间缺乏严密的依赖校验。如果前面的 TIGER 处理失败，后续步骤会因为找不到 dialog_wav 抛出二次异常，掩盖原始错误。

5. 资源泄露
-----------
- [隐患] 临时文件清理: 
  使用 NamedTemporaryFile(delete=False) 创建的对话提取文件在某些失败路径下没有被显式删除，长期使用会占用大量磁盘空间。

6. 总结与建议
-------------
项目整体架构解耦做得很好（Registry/Step 模式非常先进），但目前的具体实现细节（显存管理、线程同步）还处于“暴力开发”阶段。
建议优先修复：
1. 信号槽 UI 更新机制。
2. 更加智能的模型缓存/卸载策略。
3. 真正的流式/分块推理逻辑。
